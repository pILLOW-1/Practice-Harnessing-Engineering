# 第 1 课: 什么是 Harness

## 目标

理解为什么需要 Harness Engineering，并亲身体验"弱 harness"与"强 harness"的效果差异。

## 概念引入 (5 分钟)

### 问题: 模型强 ≠ 执行可靠

世界上最强的 AI 模型，如果没有一个合适的工作环境，依然会在真实工程任务中翻车。

**真实对照实验**（Anthropic）：

| 配置 | 模型 | 时间 | 花费 | 结果 |
|------|------|------|------|------|
| 无 harness | Opus 4.5 | 20 分钟 | $9 | 游戏核心功能跑不起来 |
| 完整 harness | Opus 4.5 | 6 小时 | $200 | 游戏可以正常游玩 |

同一个模型，同一提示词。换的是 harness，不是模型。

### Harness 是什么？

> **Harness = 模型权重之外的一切工程基础设施。**

用一个类比：

```
没有 Harness                    有 Harness
=============                   ==========

一个只有食材的厨房              一个完整的厨房
  - 有食材（强模型）               - 食材（模型）
  - 没有灶台                      - 灶台（环境）
  - 没有刀具                      - 刀具（工具）
  - 没有菜谱                      - 菜谱（指令）
  - 没有出菜检查口                - 出菜检查口（验证）
                                    - 备菜台（状态管理）

你只能用手把食材捏熟            你能按菜谱系统性地做菜
```

### 五子系统模型

Harness 由五个子系统组成：

| 子系统 | 厨房类比 | 代表文件 |
|--------|----------|----------|
| **指令** | 菜谱架 | AGENTS.md / CLAUDE.md |
| **状态** | 备菜台 | progress.md, feature_list.json |
| **验证** | 出菜检查口 | init.sh, tests |
| **范围** | 份量控制 | "一次做一个功能"的规则 |
| **生命周期** | 开关门流程 | init.sh, clean-state-checklist |

## 实践环节 (15 分钟)

### 练习 1: 体验弱 harness

在 `workspace-starter/` 中：

1. 打开这个项目目录（目前是空的，只有基本骨架）
2. 启动 Claude Code，给出任务："在这个项目中实现一个文档导入功能，支持 TXT 和 PDF"
3. **不给任何 harness 文件**，只给这段提示词
4. 观察 agent 的表现，记录：
   - 它先做了什么？
   - 它跳过了什么？
   - 结果如何？

### 练习 2: 体验最小 harness

在 `workspace-starter/` 中：

1. 添加最小 harness 文件：
   - `CLAUDE.md`（只包含开工流程和一条规则）
   - `init.sh`（包含安装和基础验证）
2. 用同样的任务再试一次
3. 对比两次结果

### 记录观察

在练习过程中回答：

1. 没有 harness 时，agent 的第一个错误是什么？
2. 有了最小 harness 后，哪一步改善了？
3. 还有哪里不够好？（这是后面几课要解决的问题）
