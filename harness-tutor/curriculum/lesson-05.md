# 第 5 课: 会话连续性

## 目标

学会编写 session-handoff.md，并实践多会话连续性——让 agent 关掉再打开还能接着干。

## 概念引入 (5 分钟)

### 问题: 长时任务为什么会丢失上下文

Agent 会话结束后，所有上下文都在聊天历史里。下一轮开始时：
- 新会话没有上轮的记忆
- 不知道上轮做了什么、卡在哪里、下一步是什么
- 可能重复做同一件事，或者做完全不同的东西

**Anthropic 的发现**：跨会话连续性是长时任务可靠性的关键瓶颈之一。进度日志（`claude-progress.md`）解决了基本的"从哪继续"问题，但长会话还需要一个更精炼的交接摘要。

### session-handoff.md vs claude-progress.md

| | claude-progress.md | session-handoff.md |
|---|---|---|
| **性质** | 持续的进度日志 | 单轮会话的交接摘要 |
| **更新频率** | 每轮都追加 | 只在较长会话结束时写 |
| **长度** | 越来越长 | 短小精炼（~30 行） |
| **受众** | 下一轮 agent 阅读 | 人或下一轮快速扫描 |
| **核心信息** | "做了什么 + 验证了什么" | "现在什么状态 + 下一步做什么 + 不要动什么" |

### 交接文件的核心结构

```
# 会话交接摘要

## 当前已验证         ← 哪些功能确认能用
## 本轮改动           ← 改了什么
## 仍损坏或未验证     ← 已知问题和风险
## 下一步最佳动作     ← 下一步 + 不要动什么
## 快速命令           ← 启动、验证、调试命令
```

## 实践环节 (25 分钟)

### 练习 1: 编写 session-handoff.md

在 `workspace-starter/` 中：

1. 创建 `session-handoff.md`
2. 模拟一个较长会话后的交接场景，填写所有段落
3. 特别注意"不要动什么"——这是防止下一轮破坏已稳定代码的关键

### 练习 2: 完整多会话演练

这是一个 3 轮会话的连续演练：

**第 1 轮**：
1. 打开 Claude Code，任务："完成 feat-001 文档导入"
2. 让 agent 按正常流程工作
3. 结束时更新所有 harness 文件（progress, feature_list, session-handoff）

**第 2 轮**：
1. 关闭 Claude Code
2. 重新打开
3. **不给任何额外提示词**——让 agent 只读 harness 文件自己决定下一步
4. 观察 agent 是否：
   - 正确选择了 feat-002（下一个未完成功能）
   - 没有重复做 feat-001
5. 结束时更新所有 harness 文件

**第 3 轮**：
1. 再关闭，再打开
2. 重复第 2 轮的观察
3. 检查 `claude-progress.md` 的会话记录是否越来越完整

### 练习 3: 评估连续性质量

完成 3 轮后回答：

1. 哪一轮 agent 没有正确读取 harness 文件？为什么？
2. `claude-progress.md` 的哪些字段最有用？哪些字段多余？
3. `session-handoff.md` 是否在第三轮仍然有帮助？

## 小结检查

**问答**:

1. `session-handoff.md` 和 `claude-progress.md` 的核心区别是什么？
2. 在多会话演练中，第二轮最重要的 harness 文件是哪个？
