# AGENTS.md

> 本仓库面向长时运行的 coding agent 工作流。目标是让每一轮会话结束后，下一个会话仍然能无猜测地继续工作。

## 开工流程

写代码前先按顺序做这些事：

1. `pwd` — 确认当前在仓库根目录。
2. 读取本文件 `AGENTS.md` 完整内容。
3. 读取 `docs/ARCHITECTURE.md` — 系统架构地图与硬规则。
4. 读取 `claude-progress.md` — 了解最新已验证状态和下一步。
5. 读取 `feature_list.json` — 选择优先级最高的未完成功能。
6. `git log --oneline -5` — 查看最近提交。
7. 运行 `./init.sh` — 验证环境健康。
8. 在开始新功能前，先跑必需的 smoke test。

如果 init.sh 失败，**先修基础状态**，不要在坏的起点上继续叠新功能。

## 工作规则

- **一次只做一个功能**：从 `feature_list.json` 选定一个，不多做，不少做。
- **验证先行**：不要声称完成，除非验证命令真的跑过。
- **不要削弱验证**：实现过程中不要悄悄改弱验证规则。
- **以文件为准**：优先依赖仓库里的持久化文件，而不是聊天记录。
- **不要越界**：除非为了解除当前 blocker 的窄范围修复，否则不要修改与当前功能无关的文件。

## 必需文件

| 文件 | 用途 |
|------|------|
| `feature_list.json` | 功能状态的唯一事实来源 |
| `claude-progress.md` | 会话进度和当前已验证状态 |
| `init.sh` | 统一的启动与验证入口 |
| `session-handoff.md` | 较长会话的交接摘要（可选） |

## 完成定义

一个功能只有在以下**全部条件**满足时才算完成：

- [ ] 目标行为已经实现
- [ ] 要求的验证真的跑过（测试 / lint / 类型检查）
- [ ] 证据记录在 `feature_list.json` 或 `claude-progress.md`
- [ ] 仓库仍然能按标准路径重新启动

## 收尾流程

结束会话前**必须**做：

1. 更新 `claude-progress.md`（做了什么、验证了什么、下一步）。
2. 更新 `feature_list.json`（当前功能的状态变更）。
3. 记录仍未解决的风险或 blocker。
4. 在工作处于安全状态后，用清晰的提交信息提交。
5. 保证下一轮会话可以直接运行 `./init.sh`。

## 验证命令

```bash
# 完整验证（推荐）
./init.sh

# 单独检查
npm install && npm run type-check && npm test && npm run lint
```

## 升级路径

| 遇到问题 | 怎么做 |
|----------|--------|
| 架构决策 | 查阅 `docs/ARCHITECTURE.md`，或询问用户 |
| 需求不明确 | 查阅 `docs/PRODUCT.md`，或询问用户 |
| 测试反复失败 | 更新进度日志，标记需要人工审查 |
| 范围有歧义 | 重新读取 `feature_list.json` 中的完成定义 |
